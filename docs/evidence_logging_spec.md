# Evidence-Rich Training Report Specification

## Goal

Add a training-history recorder that produces evidence-rich reports for an
LLM-generated decoder bias. The recorder records only the explicit robot-task
feature `capability_match`; the separate window-level bias extension is
specified in `deepseek_bias_spec.md`.

`capability_match` must reuse the project's existing ability-mask logic. The
logging path must not change masks, action sampling, rewards, or training
trajectories.

## Required Configuration

The existing Python configuration style must expose:

```python
ENABLE_EVIDENCE_LOGGING = True
EVIDENCE_LOG_INTERVAL_STEPS = 3000
EVIDENCE_OUTPUT_DIR = "./evidence_logs"
MAX_CASES_PER_REPORT = 20
MAX_CANDIDATES_PER_DECISION = 8
```

Disabling logging must disable worker-side evidence collection and
main-process file output.

## Capability Match

The implementation must share the exact contribution test used by the
original ability mask:

```python
ability = np.maximum(np.minimum(task_status, agent_abilities), 0.0)
agent_can_help = ability.sum() > 0
```

Because the original mask is binary, `capability_match` is `1.0` when the
agent can reduce the task's remaining requirement and `0.0` otherwise. Depot
actions use `null`.

## Decision And Candidate Records

Every effective training decision records:

- episode and decision IDs, global training step, current agent ID/species and
  skill vector;
- open/completed/remaining task counts, chosen task and state;
- model entropy and valid action count;
- eventual success, deadlock, makespan, average waiting time, and average
  wasted-ability ratio;
- a bounded list of candidate actions.

Each task candidate records task ID/state, validity, whether chosen, decoder
logit and probability, remaining/original requirement vectors, and
`capability_match`. Tensor values must be detached, moved to CPU, and
converted to Python scalars/lists before entering the payload.

Candidates are the deduplicated union of the chosen action, top-probability
valid actions, the highest-match action, representative valid positive-match
actions, and a better-match alternative when the chosen match is low. The
result is capped by `MAX_CANDIDATES_PER_DECISION`.

Unknown eventual outcomes are buffered and filled at episode end.

## Episode Records

Each episode records:

- `episode_id`, `train_step_start`, `train_step_end`;
- `success`, `deadlock`, `timeout`, `final_makespan`;
- `average_waiting_time`, `average_wasted_ability_ratio`, `reward`;
- agent/task/species/skill counts and task-to-agent ratio;
- maximum and average open-task counts;
- deadlock step and number of decisions.

Unavailable statistics are `null` and must not crash training.

## Process And Step Semantics

Ray workers collect and return lightweight Python payloads only. The main
process is the sole writer.

`train_step` is the main process's cumulative count of effective training
decisions/transitions received from workers. Worker job IDs remain the
existing episode counter; POMO sub-episodes receive deterministic unique
integer IDs.

## Window Files

At each configured training-step boundary, write:

```text
decisions_<start>_<end>.jsonl
episodes_<start>_<end>.jsonl
evidence_window_<start>_<end>.json
evidence_window_<start>_<end>.md
```

The final partial window must be flushable on shutdown.

The JSON report contains window metadata, global metrics, failure-mode
statistics, aggregate contrasts, representative cases, diagnosis, and the
recommended LLM output schema.

The Markdown report contains:

1. **A. Global training trend**: success/deadlock/timeout rates, average
   makespan/waiting time/wasted-ability ratio/reward/open-task count, maximum
   open-task count, and changes from the previous window.
2. **B. Failure mode distribution**: low-match decisions, low-match choices
   when a better valid alternative exists, and poor-match decisions before
   deadlock.
3. **C. Feature definition**: state that `capability_match` is binary and
   reuses the original ability-mask logic.
4. **D. Aggregate contrast**: compare decisions from the fastest 20% of
   successful episodes with failed episodes plus the slowest 20% of
   successful episodes.
5. **E. Representative failed decision cases**.
6. **F. Representative successful decision cases**.
7. **G. Current policy diagnosis**, generated by rules without calling an
   LLM.
8. **H. Recommended LLM output format**, limited to:

```json
{
  "weights": {"capability_match": 0.0},
  "lambda": 0.0,
  "clip_range": [-2.0, 2.0],
  "rationale": {
    "main_failure_modes": [],
    "expected_effect": []
  }
}
```

Default diagnosis thresholds are:

```python
chosen_capability_match < 0.3
max_candidate_capability_match - chosen_capability_match > 0.3
```

## Performance And Acceptance Requirements

- Never retain large source tensors.
- Keep writes at window frequency and write only in the main process.
- Gracefully degrade unavailable values to `null`.
- Preserve original action-mask semantics and random sampling order.
- Verify mask equivalence, JSON serialization, candidate caps, depot handling,
  timeout/deadlock classification, window boundaries, disabled logging, single
  writer ownership, fixed-seed trajectory equivalence, and minimum report
  generation.
